# 事業所防災ナビ AGENTS.md

このリポジトリは、`https://jigyousho-bousai.com/` の静的サイトを生成する。
Codex は毎回ゼロから判断しない。以下の方針を必ず継承する。

## 最上位ルール

このサイトでは、訪問者に制作過程、改善方針、AIの都合を見せない。
公開ページに出す文章は、施設担当者が購入前に判断するための情報だけにする。

公開前に必ず以下を自問する。

1. これは読者が知りたいことか。
2. 制作者の説明になっていないか。
3. 商品購入前の判断に役立つか。
4. 会社、店舗、施設担当者の言葉として自然か。
5. AIが自分の作業を説明している文章に見えないか。

1つでも怪しければ公開しない。
制作メモは AGENTS.md や作業ログに残し、HTMLには出さない。

公開ページに「編集方針」「参考サイトから反映」「改善」「生成」「AI」「UI上」「次工程」など、制作側の言葉を出したら失敗。
読者には「何を買うか」だけではなく、「何を確認してから買うか」を見せる。

## 個人情報の禁止

- 利用者がその情報の公開を明示的に指示しない限り、氏名、メールアドレス、住所、電話番号、アカウントIDなど個人を特定できる情報を公開ページへ一切入れない。
- 実名を `author`、`publisher`、運営者、著作権者、構造化データの人物として推測で追加しない。
- 表示上の主体が必要な場合はサイト名を使う。構造化データで人物・組織が必須でなければ項目自体を省略する。
- 公開前に、生成済みHTML、meta、JSON-LD、フッター、サイト情報ページを横断検索し、個人情報が残っていないことを確認する。

## サイトの目的

- 事業所、会社、店舗、保育園、介護施設、飲食店向けの防災用品比較サイトにする。
- 商品リンク集ではなく、購入前の判断を助けるサイトにする。
- 初めて来た人が「自分の施設で何を何個見るべきか」をすぐ理解できる状態にする。

## 入口設計

トップページでは商品名から入らせない。
先に以下を見せる。

- 人数
- 来客、利用者、職員数
- 待機日数
- 止まるもの: 電気、水、トイレ、交通、通信
- 置き場所と重さ
- 必要回数、容量、保存年数

トップの導線は、地震、台風、停電、断水、帰宅困難者、簡易トイレ、人数別目安を優先する。

## 絶対に出さない文

サイト訪問者に、制作側の事情を見せない。

禁止例:

- 参考サイトから反映した入口
- 診断、手順、実物感を先に出す
- TOPの上部に集めています
- UI上では
- 次工程で
- 商品取得を改善
- 取得条件の見直し対象
- AI臭い
- 編集方針
- 改善
- 生成
- AI
- 制作側、実装側、生成、プロンプト、改善方針

これらは作業メモであり、公開ページに出してはいけない。

## 文章の基準

- 読者に関係ある言葉だけを書く。
- 「何が困るか」「何を確認するか」「何を比較するか」を短く書く。
- 抽象的な宣言を避ける。
- 比較サイトの都合ではなく、施設担当者の判断順で書く。
- 「最強」「絶対」「完全対応」「これだけで大丈夫」は使わない。
- 価格、在庫、レビュー、仕様は変動するため、購入前確認の注記を入れる。

良い例:

- 人数・止まるもの・置き場所を確認
- 水、トイレ、電源、食料を順番に見る
- 仕様が分かりにくい商品は、購入前に販売ページで確認してください

## デザイン方針

- 行政資料風にしすぎない。
- AI生成テンプレート風の均一なカードだけで構成しない。
- 実物の商品画像、判断順、数量目安、比較表を上部に出す。
- 背景は薄い生成り色。
- メインカラーは濃い青緑または落ち着いたネイビー。
- アクセントは防災感のあるオレンジ。
- 赤は警告以外に多用しない。
- スマホで本文16px以上を保つ。
- スマホで横にはみ出す文章を残さない。

## 商品と比較の方針

- 楽天APIや商品取得ロジックを変更する前に、必ず現状を確認する。
- 商品数を増やす場合は、無関係な商品で水増ししない。
- 家庭用すぎる商品は事業所向けページで優先しない。
- アウトドア色だけ強く、防災・備蓄用途が弱い商品は避ける。
- 楽天の商品名をそのまま長く出さない。表示用の短い商品名を使う。

## SEO / AIO

- 各ページの冒頭に「このページの結論」を置く。
- 選び方、必要数量、比較表、商品カード、注意点、FAQを明確に分ける。
- 地震、台風、停電、断水、帰宅困難者、事業所、会社、店舗、防災備蓄、比較を自然に入れる。
- キーワードを詰め込まない。
- FAQは実際の購入前の疑問に答える。
- JSON-LDは表示している商品情報だけで作る。
- レビューがない商品にaggregateRatingを入れない。

## 参考にする考え方

競合や参考サイトを見るときは、見た目を真似しない。
以下の仕組みを抽出して、自サイトに合わせる。

- SUNDAY MOUNTAIN: 実際に困る場面から入る
- アイリスオーヤマ: 被災経験、5STEP、必要品カテゴリで信頼を作る
- 東京都備蓄ナビ: 診断導線と公的資料への接続で安心感を作る
- 防災専門店系サイト: 商品を探しやすくし、カテゴリを迷わせない

## 変更時の必須確認

実装後は必ず確認する。

- `npm.cmd run build`
- `npm.cmd run test:product-fit`
- `node --check scripts\generate-site.js`
- H1が1ページ1つ
- JSON-LDが壊れていない
- 内部リンク切れがない
- NG文言が公開HTMLに残っていない
- スマホ幅で文字が切れていない
- 公開URLに反映されたか

NG文言チェック例:

`編集方針|参考サイト|反映した|実物感|TOPの上部|集めています|改善|生成|AI|UI上|次工程|商品取得を改善|取得条件|制作|AI臭`

## 公開前の判断

公開前に自問する。

- これは訪問者が読むべき文か。
- 制作者の説明になっていないか。
- 商品を買う前の判断に役立つか。
- 会社や施設の担当者がスマホで読んでも意味が通るか。
- 「AIが説明しているページ」に見えないか。

1つでも怪しければ直してから公開する。

## 品質ゲート

このリポジトリでは、外部のエージェント基盤を増やすより、既存の検証を販売導線の品質ゲートとして固定する。

### UI変更前

- Hallmark式の読み取り専用監査を先に行う。
- 既存ブランド、情報設計、楽天API更新、自動生成を維持する。
- 架空の実績、専門家コメント、レビュー、購入数を作らない。
- 均一なカードを増やす前に、施設担当者の判断順が短くなるか確認する。

### コード変更前

軽量な影響マップを作る。

`変更ファイル -> 呼び出し元または生成先 -> 対象テスト`

商品判定を変える場合は、少なくとも以下を確認する。

- 商品取得時の候補順位
- 既存 `products.json` を使った静的生成
- 比較表、商品カード、JSON-LD
- GA4の商品ID、価格、CTA位置
- 週次更新時のフォールバック

### テストとレビュー

- 商品判定はテストを先に追加してから実装する。
- 表示変更は320px、375px、414px、768pxで確認する。
- 実装者とは別の観点で、検索意図、購買判断、誤分類、公開文言を再監査する。
- 「公開できる」だけでは合格にしない。検索流入、比較利用、楽天クリック、確定報酬で改善を判定する。
